1背景与目标

现状痛点

座舱内各业务功能分属不同厂商 APK，独立部署。模块间通信现状以 广播（~40%）+ AIDL（~30%） 为主，存在的问题：

  • 耦合高：各模块两两直接 bindService / 自定义广播 Action，新增一方要改多方。
  • 无统一身份与鉴权：广播易被伪造、嗅探；AIDL 接口各家不一。
  • 稳定性靠各自实现：对端崩溃、开机竞速、僵尸 Binder 无统一兜底，易出现永久断连 / ANR。
  • 无版本协商：接口演进破坏兼容。

设计目标

G1
低耦合
只依赖 contract 常量
G2
极致稳定
崩溃/竞速/伪造有对策
G3
自动重连
恢复后业务零感知
G4
低成本接入
适配存量 service
关键定位：无独立 broker / kernel APK，去中心化——每个集成 SDK 的 App 都是对等 Node，借 Android 系统的 bindService 作为「连接」中枢；连接目标由各 :contract:X 提供（ServiceNode 节点坐标，注入式 DI，无静态 JSON 清单），点对点建立长连接。这是相对老项目 N80 的低侵入演进方案（中心化中间件为新项目的终极方案，本期不采纳）。

2总体架构与分层

严格单向依赖：业务 App → :contract:X → :core:full / :core:lite → :transport。:transport 是唯一跨进程层，AIDL 评审通过后永不改签名。

Gradle 模块artifactId职责
:transportbridge-transport唯一 AIDL（冻结区）：IBridgeNode + BridgeEnvelope
:core:litebridge-core-lite发现/连接/重连/编解码/RPC/分发/鉴权，统一门面 Bridge，无 Service
:core:fullbridge-core= core:lite + 托管 BridgeNodeService（manifest 自动合并）
:contract:usercenter / :mediabridge-contract-*模块 topic + schema + 强类型门面，团队自治

3传输协议与核心概念

BridgeEnvelope（信封）

跨进程最小单元 = 通用 Header + JSON payload。Header 与 payload 分离：路由/去重/版本判断只读 Header，转发不反序列化 payload。

msgId          // UUID，端到端幂等去重
type           // REQUEST/RESPONSE/EVENT/HELLO/ERROR
topic          // 路由键 media.play / usercenter.account.state
schemaVersion  // 跨版本协商
source         // 发送方节点 id（接收侧不信，以 uid 校验）
correlationId  // REQUEST↔RESPONSE 关联
code           // 错误码 0=OK
payload        // JSON 字符串
ABI_VERSION=1  // 传输 ABI 版本常量（只增不改签名）

IBridgeNode（唯一 AIDL）

oneway void deliver(BridgeEnvelope env)
  // 请求/响应/事件/握手统一入口
  // oneway：投递即返回，绝不阻塞调用线程

void attach(IBridgeNode peer, String id)
  // 注册反向通道，形成全双工
  // lite 纯客户端靠它才能收到 response/event

核心概念

Node一个集成 SDK 的 App 进程，id = 包名
Topic点分路由键，前缀=模块名
HELLO能力名片：声明本端 provide/subscribe
topic 能力声明的唯一来源是 onRequest(provide) 与 subscribe(subscribe)，经 HELLO 通告对端——无需逐 topic 登记。register 只做两件事：声明模块身份（register(module)）与注入连接坐标（register(ServiceNode)），不重复声明 topic。

对端能力表（PeerConnection）与 HELLO 刷新机制

每个 PeerConnection = 一个对端 + 它自报的两张能力清单。能力一变就向所有 peer 重广播 HELLO，对端 handleHello 用 clear+add 全量替换（幂等，天然支持重连补发）。

查询：provides(topic) 精确匹配（RPC 路由）· subscribes(topic) 精确或 "module.*" 前缀（事件路由）· evaluateReady 扫 providedTopics 前缀（模块就绪）。读在调用线程、写在 worker 线程，故能力集用 CopyOnWriteArraySet。

4线程模型

线程来源职责约束
Binder 线程系统线程池deliver() 到达点，取 getCallingUid()不做业务，立即 post 给 worker
worker（单线程）SDK daemononInbound + 去重 + 鉴权 + onRequest handler 内联 + 回调串行，杜绝并发乱序；慢 handler 会队头阻塞，应改用 async 池
async-handler 池SDK daemon（懒建·固定大小）onRequestAsync 注册的 handler 在此并行执行不占 worker；handler 间并发，业务自管
scheduler（单线程）SDK daemonRPC 超时计时、退避重连不执行业务逻辑
主线程App调用 API / 更新 UI回调在 worker，UI 自行切主线程；requestSync 禁主线程
不阻塞原则：deliver 全程 oneway，对端再慢也不会占用本端 Binder 线程；同步糖 requestSync 在主线程调用直接抛 IllegalStateException 防 ANR。耗时 handler 用 onRequestAsync 走独立线程池，避免占用单线程 worker 队头阻塞；误用 onRequest 同步耗时 ≥200ms 会打告警。

5链路①：启动 / 建连 / 握手（方法级）

从 Application.onCreate 到双端互知能力的完整调用链。

Application.onCreate()
└─ Bridge.init(ctx)
│  └─ new BridgeCore(ctx)   // 建 worker/scheduler/RpcEngine/ConnectionManager/AclGuard
│  └─ BridgeCore.start()   // 只打印版本横幅，不再读任何清单
└─ .register(XxxContract.NODE, cb)   // 契约注入节点坐标（ServiceNode：pkg/action/component/module/契约版本）
   └─ BridgeCore.register(node, cb)
      ├─ registerModule(module, contractVersion, cb)   // 登记模块 + 就绪回调（onConnected/onReady/onRebooted）
      └─ ConnectionManager.connect(node)          // 按坐标 bind，已知节点去重 + self 跳过
         └─ ctx.bindService(Intent(action/component), conn, BIND_AUTO_CREATE)
            └─ 返回 false → scheduleReconnect(node)   // 退避重连
[系统拉起对端 Service → onBind 返回 Bridge.nodeBinder()]
└─ ServiceConnection.onServiceConnected(binder)
   ├─ remote = IBridgeNode.Stub.asInterface(binder); peers.put(id, new PeerConnection)
   ├─ core.linkDeath(pc)     // b.linkToDeath(DeathRecipient)
   ├─ core.attachTo(remote)  // remote.attach(localNode, selfId) —— 交出反向通道
   └─ core.sendHelloTo(pc)   // pc.send(buildHello()) —— 声明 provide/subscribe

对端：localNode.attach → onAttach（建 PeerConnection + 回握）
      localNode.deliver(HELLO) → worker → onInbound → handleHello → 存对端能力表

连接拓扑与寻址：1 对 1 / 多对 1 / 多对多

没有中心注册表：寻址 = 系统按 ServiceNode 坐标解析 Intent（阶段 A 建连）；连上后靠各 peer 的能力表按 topic 路由（阶段 B）。bind 是单向的，attach 交出反向通道后才双工。

实线 = 主动 bindService 方向；紫色虚线 = attach 交出的反向通道。连接键 = 包名；connect() 内 self 跳过 + known/peers 双重去重保证幂等。

6链路②：RPC 请求-响应（方法级）

消费方 request 一次调用，跨进程拆成两条独立 oneway 消息，由 RpcEngine 按 correlationId 缝合 + 超时兜底。

// 消费方
Bridge.request(topic, payload, reply, timeoutMs)
└─ BridgeCore.request
   ├─ findProvider(topic)  ──无──▶ reply.onError(E_NO_PROVIDER)
   ├─ corr = UUID
   ├─ rpc.register(corr, provider.peerId, reply, timeoutMs)   // 登记 + 超时计时
   └─ provider.send(newEnv REQUEST) → remote.deliver(env)  ──失败──▶ rpc.complete(corr, E_NOT_CONNECTED)

// 提供方（worker 串行）
deliver(REQUEST) → onInbound → firstSeen 去重 → acl.verifySource(uid)
└─ handleRequest
   ├─ handlers.get(topic)  ──无──▶ src.send(RESPONSE, E_NO_PROVIDER)
   └─ handler.handle(new BridgeRequest(payload), new BridgeResponder(sink))
      └─ resp.ok / resp.fail  →  sink → src.send(newEnv RESPONSE, corr, code) → deliver

// 消费方收响应
deliver(RESPONSE) → onInbound → handleResponse → rpc.complete(corr, code, payload)
   └─ remove + done.CAS(true) + 取消超时 → reply.onSuccess(payload) / onError(code, msg)

七条结束路径：全部恰好回调一次 reply

无论成功、超时、对端无 handler、对端异常、投递失败还是对端崩溃，最终都精确触发一次 onSuccess 或 onError——靠 pending.remove + done.CAS 双保险。

线程色：蓝=调用线程 · 橙=scheduler · 绿=worker。除①（无提供方，同步直返）外，所有路径都汇入 rpc.complete 的 CAS，保证 reply 恰好触发一次。

7链路③：Event 发布-订阅（方法级）

首屏时序兜底：消费方启动晚于事件发布时，用一次 request（如 usercenter.getAccount）主动拉取当前值，避免漏掉最后状态。

8入站统一管道与稳定性

统一入站管道

IBridgeNode.Stub.deliver(env)   // Binder 线程
  ├─ uid = Binder.getCallingUid()   // 内核身份
  └─ worker.execute(() -> onInbound(env, uid))
       ├─ firstSeen(msgId)      // LRU 1024 去重
       ├─ acl.verifySource(uid, source)  // 伪造即丢弃

            REQUEST  → handleRequest
            RESPONSE → handleResponse
            EVENT    → handleEvent

稳定性对策

场景机制
对端崩溃DeathRecipient.binderDied → peers.remove + rpc.failPeer(E_NOT_CONNECTED)
对端恢复BIND_AUTO_CREATE 自动回调 onServiceConnected → 重握手 + 重订阅 → 回调 onRebooted()
开机竞速 bind=falsescheduleReconnect 退避（1s→×2→30s）
身份伪造getCallingUid + PackageManager 校验，不信 source
重复投递msgId 有界 LRU 去重，幂等
慢对端拖垮deliver oneway + worker 串行隔离

9消费方实现原理 CONSUMER

消费方=发起 RPC / 订阅事件的一端（如导航订阅账号、调多媒体播放）。

  1. 依赖：:core:lite + 关心模块的 :contract:X（只用 schema 常量，不依赖 provider 实现）。
  2. 初始化（链式）：Bridge.init(this).register(XxxContract.NODE, cb) —— 注入契约提供的 ServiceNode 坐标即触发「连接 + 模块注册 + 就绪回调」；纯客户端无需暴露 Service，靠 attach 把本端 localNode 交给对端，从而能收到 response/event。
  3. 就绪跟踪：register 传入 ModuleCallback 监听 onConnected（bind 成功）/ onReady（握手完成可调用）/ onRebooted（提供方崩溃恢复）；亦可 Bridge.isReady(module) 同步查询。非就绪时调 request 会打印告警并回 E_NO_PROVIDER。
  4. 调用 RPC：Bridge.request(topic, payload, reply, timeout) —— 本地立即返回，结果异步走 reply；超时/无 provider/断连均收口到 onError。
  5. 订阅事件：subscribe / subscribes(t..).on / subscribeAll(module).on —— 订阅声明经 HELLO 告知 provider，provider 据此定向推送。
  6. 强类型门面（可选）：UserCenterClient.subscribeAccountState(...) 封装 topic 字符串，调用方零记忆负担。
public class NaviApp extends Application {
  public void onCreate() {
    super.onCreate();
    Bridge.init(this)                                       // 链式：init 返回 BridgeSetup
      .register(UserCenterContract.NODE, moduleCb("账号"))     // 注入账号节点坐标 + 就绪回调
      .register(MediaContract.NODE,      moduleCb("多媒体"))   // 注入多媒体节点坐标 + 就绪回调
      .subscribes(UserCenterSchema.ACCOUNT_STATE, MediaSchema.STATE)
      .on((topic, payload) -> push(topic + ": " + payload));   // 一个回调收多类状态
  }
  static ModuleCallback moduleCb(String name) {                // onConnected/onReady/onRebooted
    return new ModuleCallback() {
      public void onReady()   { push("【"+name+"】就绪可调用"); }
      public void onRebooted(){ push("【"+name+"】提供方重启恢复"); }
    };
  }
  static void pullAccount() {
    UserCenterClient.getAccount(new BridgeReply(){              // = Bridge.request 首屏兜底
      public void onSuccess(String p){ ... } public void onError(int c,String m){ ... }
    }, 3000);
  }
}

10提供方实现原理 PROVIDER

提供方=响应 RPC / 发布事件的一端（如用户中心响应 getAccount、发布登录态）。

  1. 暴露通道（二选一）：
    • 全量 依赖 :core:full，BridgeNodeService 经 manifest 自动合并，被 bind 时 onBind 返回 Bridge.nodeBinder()——业务无感知。
    • 外挂 依赖 :core:lite，存量 App 在已有 Service 的 onBind 里按 action 返回 Bridge.nodeBinder()——不新增 Service 类、不增进程。
  2. 注册能力：Bridge.onRequest(topic, handler)（handler 在 worker 串行内联，适合快 handler）或 Bridge.onRequestAsync(topic, handler)（耗时 handler 在独立线程池并行，不占 worker、不队头阻塞）声明可响应的 request topic → 经 HELLO 通告对端 provide 列表（async 对对端透明）。
  3. 响应请求：收到 REQUEST → handler.handle(BridgeRequest, BridgeResponder) → resp.ok(json) / resp.fail(code, msg)（一次性 CAS，防重复应答乱序）。耗时处理务必用 onRequestAsync；误用 onRequest 同步占 worker ≥200ms，SDK 打告警引导改用 async。
  4. 发布事件：Bridge.publish(topic, payload) → 按对端订阅声明（含 module.* 前缀）定向推送。
  5. 身份安全：一律 getCallingUid() 校验真实包名，绝不信 Envelope.source。
public class UserCenterApp extends Application {   // 外挂形态
  public void onCreate() {
    super.onCreate();
    Bridge.init(this);
    Bridge.onRequest(UserCenterSchema.GET_ACCOUNT, (req, resp) -> resp.ok(currentAccount));
  }
  static void updateAccount(...) { ...; Bridge.publish(UserCenterSchema.ACCOUNT_STATE, json); }
}
public class HostService extends Service {        // 复用宿主已有 Service
  public IBinder onBind(Intent i){
    if (BRIDGE_ACTION.equals(i.getAction())) return Bridge.nodeBinder();
    return mHostOwnBinder;
  }
}

12关键设计取舍

选择理由
去中心化（无 broker）无预装特权、无需常驻中心进程；借系统 bindService 做发现，少一个崩溃点。
deliver 全 oneway跨进程绝不同步阻塞；请求-响应靠 correlationId + 超时缝合，杜绝慢对端拖垮线程池 / ANR。
Header / payload 分离 + JSON路由不反序列化 payload；跨版本字段增删容忍度高，配合 schemaVersion 兼容。
AIDL 冻结跨进程 ABI 稳定，演进只在 topic/schema/payload，避免双端版本地狱。
契约自治SDK 只给内核+模板+规范，各模块在自己的 contract 补 topic/错误码/门面，全局只维护前缀+错误码区间分配表。
连接坐标契约注入（DI）坐标由 :contract:X 的 ServiceNode 提供、消费方组合根注入，替代静态 JSON 清单：漏依赖 = 编译错，零反射零 IO，core 永不反向依赖 contract。
topic 能力 = onRequest/subscribetopic 收发能力的唯一来源，无需逐 topic 登记；register 仅声明模块身份与注入连接坐标。
onRequest 内联 + onRequestAsync 池快 handler 在单线程 worker 串行（无锁、简单）；耗时 handler 走独立线程池避免队头阻塞——同一 API 风格两条路，误用同步耗时有运行时告警兜底。
当前状态：全量编译通过（5 aar + 3 示例 apk），纯逻辑单测全绿。已在手机 + Root 车机台架双设备验证：契约注入式连接（无 JSON）、media.play（RPC）+ account.state（Event）全链路、kill -9 杀提供方后 ~1s 自愈并回调 onRebooted 的闭环；以及 onRequestAsync 慢 handler（2s）不阻塞同期快请求（+6ms 返回）、onRequest 同步耗时 302ms 触发预警告警。
