# Bridge SDK：模块状态回调 + 非就绪告警 + 版本/构建信息日志 设计

- 日期：2026-06-23
- 范围：`:core:lite`（主）、`:transport`（加常量）、节点静态清单格式扩展
- 不触碰：`:transport` 的 AIDL（`IBridgeNode`/`BridgeEnvelope`）方法签名（冻结区）

## 1. 背景与目标

Bridge SDK 是去中心化点对点总线，无中心进程。一个「模块」= topic 前缀（如
`usercenter`/`media`）；消费方 `Bridge.register(module)` 声明关心某模块；对端完成
HELLO 握手后，`PeerConnection.providedTopics` 才持有该模块的 topic。

本次新增三个能力：

1. `register(module, callback)`：模块状态回调，提供 `onConnected()` / `onReady()` /
   `onRebooted()`，并支持 `isReady(module)` 查询。
2. 非就绪状态调用 `request` 时打印告警日志，便于排查。
3. 启动日志打印 SDK 版本、契约门面版本、git commit hash 等信息。

## 2. 关键语义决策（已与需求方确认）

| 决策点 | 结论 |
|---|---|
| 「ready」定义 | 存在「已连接 + 完成 HELLO 握手 + `providedTopics` 含该模块前缀」的对端 |
| `onConnected` | 提供该模块的节点 `bindService` 成功（`onServiceConnected`，**尚未握手**），主要用于排查日志 |
| `onConnected` 归属 | 节点静态清单新增 `module→节点` 映射（bind 时只知包名，无法靠握手推断） |
| `onRebooted` | 模块曾 ready，其提供方对端断开（`binderDied`/`onServiceDisconnected`）后，重连并重新握手完成时触发 |
| 契约门面版本 | 由各 contract 门面把自己的 `Schema.VERSION` 通过 `register(module, version, cb)` 传入，core 记录并打印（铁律：core 不依赖 contract） |
| 回调线程 | 统一在 `bridge-worker` 单线程串行触发；更新 UI 由业务自行切主线程（与 `request` 的 reply 一致） |
| 版本信息来源 | `BuildConfig`（git hash 构建时自动注入），而非手写常量 |

## 3. 模块映射（前置改动）

节点静态清单 `assets/bridge_nodes.json` 每个节点增加**可选** `modules` 字段：

```json
{
  "version": 1,
  "nodes": [
    { "id": "com.baic.usercenter", "action": "com.baic.usercenter.HOST",
      "component": "com.baic.usercenter/.HostService", "modules": ["usercenter"] },
    { "id": "com.baic.media", "action": "com.baic.bridge.NODE", "modules": ["media"] }
  ]
}
```

- `NodeDescriptor` 增字段 `final java.util.Set<String> modules`（缺省空集，向后兼容）。
- `NodeRegistry.load` 解析 `o.optJSONArray("modules")`。
- `BridgeCore` 据此建 `Map<String nodeId, Set<String> modules>`，**仅用于 `onConnected` 归属**。
  `onReady`/`onRebooted`/`isReady` 仍以握手得到的 `providedTopics` 前缀匹配为准，与该映射无关。

> 兼容性：未声明 `modules` 的节点，其 `onConnected` 不触发（拿不到归属），但
> `onReady`/`onRebooted` 不受影响——握手后即可正确判定就绪。

## 4. 特性一：模块状态回调 + isReady

### 4.1 新增接口 `ModuleCallback`（`:core:lite`）

```java
package com.baic.bridge.core;

/** 模块状态回调。所有方法在 SDK worker 线程串行触发，更新 UI 请自行切主线程。 */
public interface ModuleCallback {
    /** 提供该模块的节点 bindService 成功（传输通道建立，尚未握手）。排查日志用。 */
    default void onConnected() {}
    /** 提供方握手完成、该模块能力首次可用。 */
    void onReady();
    /** 提供方重启恢复（曾 ready → 断开 → 重连重握手）后再次可用。 */
    default void onRebooted() {}
}
```

### 4.2 门面 `Bridge` 新增 API（保留旧 `register(String)` 不破坏兼容）

```java
public static void register(String module);                                   // 既有
public static void register(String module, ModuleCallback cb);                 // 新增：契约版本未知(0)
public static void register(String module, int contractVersion, ModuleCallback cb); // 新增：携带契约门面版本
public static boolean isReady(String module);                                  // 新增
```

### 4.3 `BridgeCore` 就绪状态机

每个已注册模块维护一条 `ModuleState`：

```
class ModuleState {
    final String module;
    final int contractVersion;          // 仅用于日志
    final CopyOnWriteArrayList<ModuleCallback> callbacks; // 同一模块可多次 register
    volatile boolean readyOnce;         // 历史上是否就绪过
    volatile boolean currentlyReady;    // 当前是否有握手完成的提供方
}
```

存储：`ConcurrentHashMap<String module, ModuleState>`。

判定函数 `boolean evaluateReady(module)` = 遍历 `connections.peers`，存在某对端
`providedTopics` 含以 `module + "."` 为前缀的 topic 即为 true。

重算 `reevaluate(module)`（**worker 线程执行**）：

```
boolean now = evaluateReady(module);
if (!state.currentlyReady && now) {
    state.currentlyReady = true;
    if (!state.readyOnce) { state.readyOnce = true; fire onReady(); log "模块 X 就绪"; }
    else                  { fire onRebooted();      log "模块 X 提供方重启恢复"; }
} else if (state.currentlyReady && !now) {
    state.currentlyReady = false;       // 无回调（未要求 onLost）
    log "模块 X 提供方已离线";
}
```

`isReady(module)` 返回 `state.currentlyReady`（未注册模块返回 false）。

### 4.4 触发点与线程

| 事件 | 现状位置 | 改动 |
|---|---|---|
| 握手完成 | `BridgeCore.handleHello`（worker） | 末尾对所有已注册模块 `reevaluate`（providedTopics 已更新） |
| 对端死亡 | `linkDeath().binderDied`（binder 线程） | `worker.execute(() -> { peers.remove; rpc.failPeer; reevaluateAll(); })` |
| 对端断开 | `ConnectionManager.onServiceDisconnected`（主线程） | 改为回调 `core.onPeerLost(nodeId)`，内部 `worker.execute` 移除并 `reevaluateAll` |
| bind 成功 | `ConnectionManager.onServiceConnected`（主线程） | 新增回调 `core.onPeerConnected(nodeId)`，内部 `worker.execute`：查 `节点→modules`，对已注册模块触发 `onConnected()` |

> 所有回调与状态变更统一 post 到 `bridge-worker` 串行执行，避免并发竞态（符合铁律）。

## 5. 特性二：非就绪调用 request 告警日志

`BridgeCore.request()` 中 `findProvider(topic) == null` 分支，在返回 `E_NO_PROVIDER`
前新增告警：

```java
Log.w(TAG, "request 调用时模块未就绪 topic=" + topic
        + " module=" + moduleOf(topic)
        + " isReady=" + isReady(moduleOf(topic))
        + "，当前无提供方，返回 E_NO_PROVIDER；建议等 onReady 回调后再调用");
```

`moduleOf(topic)`：取首个 `.` 前的前缀；无 `.` 则返回原串。行为不变（仍走原
`reply.onError(E_NO_PROVIDER)`），仅补日志。

## 6. 特性三：启动版本/构建信息日志

### 6.1 构建注入（`:core:lite/build.gradle.kts`）

```kotlin
android {
    buildFeatures { buildConfig = true }
    defaultConfig {
        minSdk = 28
        buildConfigField("String", "SDK_VERSION", "\"${'$'}sdkVersion\"")
        buildConfigField("String", "GIT_SHA", "\"${'$'}gitSha\"")
        buildConfigField("String", "BUILD_TIME", "\"${'$'}buildTime\"")
    }
}
```

- `sdkVersion`：常量（首版 `1.0.0`）。
- `gitSha`：`git rev-parse --short HEAD` 的输出，捕获异常兜底 `"unknown"`（无 git 环境不报错）。
- `buildTime`：ISO-8601 构建时间戳。
- 生成的类为 `com.baic.bridge.core.BuildConfig`。

### 6.2 传输 ABI 常量（`:transport`）

`BridgeEnvelope` 增 `public static final int ABI_VERSION = 1;`（仅新增常量，不改方法签名，不违反冻结）。

### 6.3 启动横幅（`BridgeCore.start()`）

```
I/Bridge.Core: ==== Bridge SDK 启动 ==== self=com.baic.navi
I/Bridge.Core: SDK版本=1.0.0 gitSha=250fd7c 构建=2026-06-23T10:00:00Z 传输ABI=1 清单节点数=2
```

### 6.4 契约门面版本日志（`register(module, version, cb)` 时）

```
I/Bridge.Core: 注册模块 usercenter 契约门面版本=1 callback=true
```

契约门面（如 `UserCenterClient`）后续可提供 `register(ModuleCallback)` 便捷方法，内部调用
`Bridge.register(MODULE, UserCenterSchema.VERSION, cb)`，把版本传入。本设计不强制改 contract，
仅打通通道。

## 8. 测试与验证

- 编译：`export JAVA_HOME=~/Library/Java/JavaVirtualMachines/liberica-17.0.18 && ./gradlew assembleDebug`。
- 真机/多 App 验证：
  1. navi-consumer 先启动，account/media provider 后启动 → 观察 `onConnected → onReady` 顺序与日志。
  2. 杀掉并重启 provider → 观察 `onRebooted` 触发、`isReady` 在断开期间为 false。
  3. provider 未起时调用 `request` → 观察非就绪告警日志 + `E_NO_PROVIDER`。
  4. App 启动日志含 SDK 版本/gitSha/构建时间/契约门面版本。

## 9. 非目标（YAGNI）

- 不提供 `onLost`/`unregister` 回调（本次未要求）。
- 不引入 module→node 的运行时动态发现（仅静态清单）。
- 不改 AIDL，不引入 ack/重传等传输层增强。
