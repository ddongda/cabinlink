# Bridge SDK 接入指南

> 车机座舱**去中心化**跨进程通信 SDK。本指南面向**第三方提供方 / 消费方开发者**，按角色照抄即可接入。
> 全景与原理见 [README](../README.md) 与 [架构汇报](Bridge-SDK-架构汇报.html)。

---

## 0. 30 秒选型

| 你的模块要… | 角色 | 依赖 aar | 暴露 Service |
|---|---|---|---|
| 被别人调用（响应 RPC / 发布事件） | **Provider** | `bridge-core`（全量，自带 Service） | 自动 |
| 同上，但已有自己的 Service 想复用 | Provider | `bridge-core-lite` | 在已有 `Service.onBind` 返回 `nodeBinder()` |
| 只调用别人、不被调 | **Consumer** | `bridge-core-lite` | 否（纯客户端） |

> 一个模块可**同时**是 Provider 和 Consumer（既被调又调别人）——按“要被调”取全量 aar 即可。
> 唯一初始化入口都是 `Bridge.init(context)`，在 `Application.onCreate` 调用。

## 1. Consumer 接入（调用别人）

**① 依赖**（以依赖“多媒体”为例）：
```kotlin
implementation(project(":core:lite"))
implementation(project(":contract:media"))   // 提供 MediaContract.NODE + topic/schema 常量
```

**② 初始化 + 注入依赖节点**（`Application.onCreate`）：
```java
Bridge.init(this)
      .register(MediaContract.NODE, new ModuleCallback() {
          public void onConnected() {}              // 已连上（未必就绪）
          public void onReady()     { /* 可安全调用了 */ }
          public void onRebooted()  { /* 提供方重启恢复，宜重新拉状态 */ }
      });
```

**③ 调用（RPC）/ 订阅（Event）**：
```java
// RPC：异步（推荐）。回调在 SDK worker 线程，更新 UI 自行切主线程
Bridge.request(MediaSchema.PLAY, "{\"trackId\":\"1001\"}", new BridgeReply() {
    public void onSuccess(String payload) {}
    public void onError(int code, String msg) {}
}, 3000);

// 订阅事件
Bridge.subscribe(MediaSchema.STATE, payload -> { /* ... */ });
```
> 调用前可 `Bridge.isReady(MediaSchema.MODULE)` 查就绪；未就绪调用立即回 `E_NO_PROVIDER`，不空等。
> 也可用 contract 自带门面 `MediaClient.play(...)` 省去 payload 拼装。

## 2. Provider 接入（被别人调用）

**① 依赖**（全量形态，自带 Service）：
```kotlin
implementation(project(":core:full"))
implementation(project(":contract:media"))
```

**② 初始化 + 声明模块 + 注册处理器**（`Application.onCreate`）：
```java
Bridge.init(this);
Bridge.register(MediaSchema.MODULE);   // 声明自身模块（不主动连别人）

// 快 handler：在单线程 worker 串行内联（默认）
Bridge.onRequest(MediaSchema.NEXT, (req, resp) -> resp.ok(stateJson()));

// 耗时 handler（查库 / 算路 / 解码）：用 async，独立线程池并行，不阻塞其它请求
Bridge.onRequestAsync(MediaSchema.PLAY, (req, resp) -> {
    String r = decodeAndPlay(req.get(MediaSchema.K_TRACK_ID));
    resp.ok(r);
});

// 发布事件给所有订阅者
Bridge.publish(MediaSchema.STATE, stateJson());
```
> **挂宿主已有 Service**（lite 形态）：依赖 `bridge-core-lite`，在你已有的 `Service.onBind` 里按 action 返回
> `Bridge.nodeBinder()`，其余写法相同——不新增 Service 类、不增进程。

## 3. 写你自己的 contract

复制脚手架 [`:contract:template`](../contract/template)，四件套改名即可：

| 文件 | 改什么 |
|---|---|
| `XxxSchema` | `MODULE`（模块前缀，全局唯一）、topic（必须 `<module>.xxx`）、payload 字段（`K_` 前缀） |
| `XxxContract` | `NODE` 坐标：pkg / action / component（全量=`null`，挂宿主=`"pkg/.Service"`） |
| `XxxError` | 错误码区间（≥1000，不与其它模块重叠）；无特殊错误码可删此类 |
| `XxxClient` | 可选门面，**薄封装**（只拼 payload + 转发，禁塞业务逻辑/状态） |

**接入登记 + 自动校验**：在 `:contract-verify` 的 `build.gradle.kts` 加一行依赖、`ContractVerifyTest.CONTRACTS` 加一行，
跑 `./gradlew :contract-verify:testDebugUnitTest`——自动校验命名 / 错误码 / NODE 一致 / 跨模块冲突，**违规即构建失败**。

## 3.5 Schema 大了怎么拆（大模块）

`XxxSchema` 把 MODULE / topic / 字段放一个类——**小模块（十几个 topic）就这样，别拆**：Schema 是纯常量声明，膨胀的是**数据量**不是**逻辑复杂度**，IDE 可折叠 / 搜索。当单类超 **~30 topic / ~200 行**、或出现 **2 个以上子域**（如车控的空调 / 座椅 / 车窗），再按子域拆：

**① 嵌套子类（首选，一个文件内分层）**：
```java
public final class HvacSchema {
    public static final String MODULE = "hvac";
    public static final class Temp {                 // 子域：温度
        public static final String SET = "hvac.temp.set";
        public static final String K_VALUE = "value";
    }
    public static final class Seat { /* ... */ }     // 子域：座椅
}
```
引用 `HvacSchema.Temp.SET`——一个 import、子域清晰、IDE 可折叠。

**② 多 Schema 文件**（子域大到要独立演进）：`HvacTempSchema` / `HvacSeatSchema`…，`MODULE` 前缀仍统一 `hvac.`；门面 `Client` 同理可拆 `HvacTempClient`。

**约束不变**：无论怎么拆，topic 仍 `<module>.xxx`、字段仍 `K_` 前缀、MODULE 前缀全局唯一——`:contract-verify` 会**递归**校验嵌套类，照样拦违规。脚手架 [`:contract:template`](../contract/template) 的 `TemplateSchema.Sub` 即嵌套分组示例。

## 4. 避坑速查（接入者必读）

| ❌ 别这样 | ✅ 要这样 |
|---|---|
| 主线程调 `requestSync` | 主线程用异步 `request`（`requestSync` 在主线程会抛异常防 ANR） |
| 在 `onRequest` handler 里同步查库 / 算路 | 耗时改用 `onRequestAsync`（否则队头阻塞；SDK 会打告警，过载回 `E_BUSY`） |
| 在 handler 里 `Binder.getCallingUid()` 自鉴权 | 不需要——SDK 已在入站校验；handler 跑在 worker / 线程池，取不到真实调用方 |
| payload 裸写字符串 key | 用 Schema 的 `K_` 字段常量（双端共用，防拼写漂移） |
| `:contract` 里依赖其它 contract / 业务实现 | contract 只依赖 `:core:lite`，互不依赖（`:contract-verify` 会拦） |
| 假设 RPC 一定成功 | 处理 `onError`（`E_NO_PROVIDER` / `E_TIMEOUT` / `E_BUSY` …），并监听 `onReady`/`onRebooted` |

## 5. 速查表

**错误码**：`0=OK`；`1–999` SDK 保留（`E_TIMEOUT=1 / E_NO_PROVIDER=2 / E_NOT_CONNECTED=3 / E_VERSION=4 / E_ACL=5 / E_INTERNAL=6 / E_BUSY=7`）；`≥1000` 各模块在自己 contract 定义。

**核心 API**：`Bridge.` `init` · `register` · `onRequest` · `onRequestAsync` · `publish` · `request` · `requestSync` · `subscribe` · `subscribes` · `subscribeAll` · `isReady` · `nodeBinder`。

**线程**：所有回调（`onSuccess`/`onError`/`onEvent`/`onReady`…）在 SDK **worker 线程**，更新 UI 自行切主线程。

**延伸**：[README](../README.md)（全景）· [架构汇报](Bridge-SDK-架构汇报.html)（方法级原理）· [`:contract:media`](../contract/media)（黄金范例）。
